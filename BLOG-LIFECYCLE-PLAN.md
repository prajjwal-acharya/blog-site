# Blog Creation Lifecycle Overhaul â€” Implementation Plan

## Context

The blog pipeline (ArticleML â†’ convert.mjs â†’ page.mdx â†’ MDXRemote rendering) has **critical rendering bugs** and a **manual, fragile component workflow**. Math renders as unstyled text (KaTeX CSS missing), GFM tables don't parse in MDX (remark-gfm missing from pipeline), code has no syntax highlighting (hljs CSS missing), and every new post component requires hardcoded imports in the blog page. The goal is a polished, simplified creation lifecycle where you write `.articleml`, run one command, drop components in the folder, and everything works.

## Phase 1: Fix Critical Rendering Bugs

These are broken **right now** for every visitor. Small, independent fixes.

### 1.1 â€” KaTeX CSS import
**File:** `src/app/layout.tsx`

Add in `<head>`:
```html
<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/katex@0.16.44/dist/katex.min.css" />
```
Math equations (`$$...$$` and `$...$`) are currently unstyled â€” rehype-katex generates the correct DOM nodes, but without this stylesheet they look like broken text.

### 1.2 â€” highlight.js CSS import
**File:** `src/app/layout.tsx`

Add in `<head>`:
```html
<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/highlight.js@11/styles/github-dark.min.css" />
```
rehype-highlight adds highlight.js classes to code blocks, but no theme CSS exists. The existing `<pre>` override already uses a dark background, so a dark hljs theme is appropriate.

### 1.3 â€” Add remark-gfm to MDX pipeline
**File:** `src/app/blog/[slug]/page.tsx`

The current MDX options only have `remarkMath`:
```ts
remarkPlugins: [remarkMath]
```
GFM tables written as pipe-tables in `.mdx` files are NOT being parsed. Fix:
```ts
import remarkGfm from "remark-gfm";
// ...
remarkPlugins: [remarkMath, remarkGfm]
```

### 1.4 â€” Fix escapeForJSX
**File:** `convert.mjs` (line 108-110)

Current implementation only escapes `\` and `"`. Add escaping for:
- `{` â†’ `&#123;`
- `}` â†’ `&#125;`
- `<` â†’ `&lt;`
- `>` â†’ `&gt;`

Without this, Callout titles containing `{` or `<` break JSX parsing.

### 1.5 â€” Rewrite processTableSegment to emit GFM markdown
**File:** `convert.mjs` (lines 289-316)

Currently outputs raw `<table>` HTML, which:
- Bypasses the styled MDX component overrides for `table/th/td` in page.tsx
- Doesn't HTML-escape cell content (cells with `<`, `&` break rendering)

Rewrite to emit standard GFM pipe-table markdown:
```markdown
| Column A | Column B |
| -------- | -------- |
| cell     | cell     |
```
This way MDX parses it via remark-gfm â†’ component overrides apply consistent styling.

### 1.6 â€” Fix placeholder regex
**File:** `convert.mjs` (line 259)

Change `[^"]*` to `(?:[^"\\]|\\.)*` so descriptions containing apostrophes/escaped quotes don't silently fail to match.

## Phase 2: Component Auto-Discovery

The core architectural change. Currently every post component is hardcoded in `page.tsx` (3 imports + 3 entries in mdxComponents). This doesn't scale.

### 2.1 â€” New file: `src/lib/load-post-components.ts`

A server-side utility function:
```ts
export async function getPostComponents(slug: string): Promise<Record<string, React.ComponentType>>
```
- Checks if `src/content/posts/{slug}/components.tsx` exists via `fs.existsSync`
- If yes, dynamically `import()` it at server render time
- Collects all named exports as components (PascalCase convention)
- Returns `{}` if no components.tsx exists

**Why this works:** MDXRemote runs in an async server component. `import()` in RSC resolves at build/render time via Turbopack, not at client runtime.

**Fallback plan:** If Turbopack chokes on variable-path imports, generate a static registry file `src/lib/__component-registry.ts` that maps slugs â†’ static imports (regenerated by the scaffold script).

### 2.2 â€” Update `src/app/blog/[slug]/page.tsx`

- **Remove** the 3 hardcoded component imports (RecomputationSim, AttentionFlow, MemoryLayersDiagram)
- **Remove** their 3 entries from `mdxComponents`
- In the async `ArticlePage` function:
  ```ts
  const postComponents = await getPostComponents(slug);
  const allComponents = { ...mdxComponents, ...postComponents };
  ```
- Pass `allComponents` to `<MDXRemote components={allComponents} />`
- Now ANY post can bring its own components just by having a `components.tsx`

### 2.3 â€” Convention for post components

Each post's `components.tsx` should:
- Use `"use client"` directive (interactive components need it)
- Export named components matching the JSX tags in the post's page.mdx
- The export name IS the JSX tag name:
  - `export function MyChart() { ... }` â†’ `<MyChart />` works in MDX
  - `export default function RecomputationSim() { ... }` â†’ `<RecomputationSim />`

### 2.4 â€” Fix `src/content/posts/memory-llm-1/components.tsx`

Currently imports `zustand` which is not in `package.json`. Replace the zustand store with React `useState`. This file becomes the canonical working example of the pattern.

## Phase 3: Converter & Workflow Improvements

### 3.1 â€” Auto-slug from frontmatter title
**File:** `convert.mjs`

When no `--slug` flag AND no explicit output path:
1. Check for `slug:` field in frontmatter (highest priority)
2. Derive from `title`: `"What Memory Really Means in LLMs"` â†’ `what-memory-really-means-in-llms`
3. Fall back to filename-based slug (current behavior)

### 3.2 â€” Placeholder â†’ Component bridging
**File:** `convert.mjs`

Extend placeholder syntax with optional `component` field:
```
[!SIMULATION] : {desc: "...", component: "RecomputationSim"}
```
When `component` is specified, output `<RecomputationSim />` directly instead of `<Placeholder>`. When omitted, output `<Placeholder>` as before (it's a scaffold for content not yet implemented).

### 3.3 â€” New post scaffold script
**New file:** `scripts/new-post.mjs`

Usage:
```bash
npm run new-post -- "My Article Title" --category "AI Advances"
```
- Derives slug from title
- Creates `src/content/posts/{slug}/` directory
- Creates a template `.articleml` file with frontmatter pre-filled (title, date, category, tags placeholder)
- Optionally creates a `components.tsx` stub with a sample export
- Add to package.json scripts: `"new-post": "node scripts/new-post.mjs"`

### 3.4 â€” Display math preprocessing
**File:** `convert.mjs`

The existing `preprocessInlineMath` handles `\(...\)` â†’ `$...$`. Add `\[...\]` â†’ `$$...$$` conversion for display math. Process display math first (greedy) to avoid conflicts with inline.

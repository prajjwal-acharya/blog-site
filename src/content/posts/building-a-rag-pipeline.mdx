Retrieval-Augmented Generation sounds clean in blog posts and messy in production. Here's what actually happens when you try to build one that works reliably.

## The Standard Pipeline

The happy-path RAG pipeline looks like this:

1. **Ingest** — chunk documents, embed them, store in a vector DB
2. **Retrieve** — embed the query, find nearest neighbors
3. **Generate** — stuff the retrieved chunks into an LLM prompt

In practice, each of these steps has failure modes that aren't obvious until you hit them.

## Chunking: The Underrated Decision

Naive fixed-size chunking (e.g., 512 tokens with 50-token overlap) destroys semantic units. A paragraph about *attention mechanisms* might be split mid-sentence, leaving each chunk missing context.

**What works better:**

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=100,
    separators=["\n\n", "\n", ". ", " ", ""],
)
```

The recursive strategy tries semantic boundaries first (double newlines = paragraph break) before falling back to sentence, then word splits. For technical docs with code blocks, you want to preserve those entirely.

## Embedding Model Choice

Not all embeddings are equal for retrieval. I tested several on a domain-specific corpus:

| Model | NDCG@10 | Latency (ms) | Cost/1M tokens |
|-------|---------|------|------|
| text-embedding-3-small | 0.71 | 12 | $0.02 |
| text-embedding-3-large | 0.79 | 25 | $0.13 |
| bge-large-en-v1.5 (local) | 0.77 | 8 | $0 |

For most use cases, `bge-large-en-v1.5` running locally is the best trade-off — nearly as good as OpenAI's large model, zero cost, and faster.

<Callout title="Key Insight">
Domain-adapted embeddings consistently outperform general-purpose ones. If your corpus has specialized vocabulary, fine-tune on a small labeled retrieval dataset (even 500 pairs helps significantly).
</Callout>

## Re-ranking Changes Everything

Top-k retrieval by cosine similarity is a blunt instrument. A cross-encoder re-ranker takes the (query, chunk) pair and scores them jointly — much more accurate, but expensive to run on all k results.

The pattern that works:

```python
# Retrieve broadly, then re-rank narrowly
retrieved = vector_store.search(query, top_k=20)
reranked  = cross_encoder.rank(query, retrieved, top_k=5)
```

In my benchmarks, re-ranking with `cross-encoder/ms-marco-MiniLM-L-6-v2` improved answer quality by ~18% on complex multi-hop questions.

## The Failure Modes

**1. Semantic mismatch**: The user asks "how does auth work?" but the docs say "authentication flow." Dense retrieval helps here; keyword BM25 search doesn't. Hybrid search (dense + sparse, combined with RRF) handles both.

**2. Lost-in-the-middle**: LLMs attend less to chunks in the middle of long contexts. Put the most relevant chunks at the beginning and end of your prompt.

**3. Stale embeddings**: If you re-embed only new documents, your vector space shifts subtly over time. Periodically re-embed everything when upgrading embedding models.

The honest take: RAG works well when the answer is *in* the documents and the query is *specific*. For broad, reasoning-heavy questions, you're better off with a long-context model that can see everything at once.
